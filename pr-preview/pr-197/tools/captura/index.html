About Captura Web Recorder

A browser-based screen recorder inspired by the original Captura desktop app. Record your screen, mix in webcam and microphone audio, and save the result — all entirely inside your browser. No data is ever sent to any server.

Limitations vs. native desktop apps
  • No global keystroke/key overlay — browsers deliberately sandbox each tab and cannot listen to keyboard events outside the browser window. OS-level keyboard hooks (used by the original Captura to show keystrokes on screen) are simply not available to web pages. This is a fundamental security boundary of the web platform, and one of the reasons native desktop apps still shine for certain capture workflows.
  • Desktop only — mobile browsers do not expose getDisplayMedia because the mobile OS sandbox prevents web pages from capturing other apps' screens. Use Chrome or Edge on a desktop.
  • Codec support varies — WebM/VP9 is widely supported; MP4/H.264 depends on the browser. The recorder automatically selects the best available codec.
  • File System Access API required — the recorder streams video directly to disk using the File System Access API. This requires Chrome or Edge; Firefox does not support this API and cannot be used to record.
Privacy: This tool runs 100% in your browser. No video, audio, or metadata is ever uploaded or transmitted anywhere.
How It Works

A look under the hood for the technically curious.

Screen & Window Capture

Screen capture uses the browser's navigator.mediaDevices.getDisplayMedia() API. When you click Start Recording, the browser shows its own native picker where you choose what to share — an entire screen, a specific window, or a browser tab. The browser returns a MediaStream containing a video track (and optionally an audio track if "Share system audio" is checked in the picker). This is intentional: the browser is the gatekeeper, so no web page can silently capture your screen without your explicit permission each time.

The system-audio checkbox in the recorder UI just passes audio: true as a constraint to getDisplayMedia. Whether system audio is actually available depends on the OS — Windows supports it natively; macOS requires a virtual audio driver.

Canvas Compositor

Rather than recording the screen stream directly, the recorder draws every frame onto an offscreen <canvas> element. This intermediate step is what makes the webcam picture-in-picture (PiP) overlay possible: each frame, the compositor first draws the screen video, then draws the webcam video on top in the chosen corner, and finally stamps a live timestamp in the bottom-right.

During recording the compositor runs on setInterval at your chosen frame rate (e.g. every ~33 ms for 30 fps). setInterval is used — rather than requestAnimationFrame — so that compositing continues reliably even when the browser tab is hidden or you switch to another application. requestAnimationFrame is throttled or paused by the browser for invisible tabs, which is why it was causing the "stuck video" bug when switching away from the page.

The canvas stream is obtained with canvas.captureStream(fps), which exposes the canvas contents as a live MediaStream video track ready to be fed into the MediaRecorder.

Audio Mixing

System audio (from the screen capture) and microphone audio are two separate MediaStreams. To combine them into a single audio track, the recorder uses the Web Audio API:

  1. An AudioContext is created.
  2. Each source stream is wrapped in a MediaStreamSourceNode.
  3. Both source nodes are connected to a single MediaStreamDestinationNode.
  4. The destination node exposes a mixed MediaStream whose audio track is then added to the canvas stream before it reaches the MediaRecorder.

This approach lets the browser's audio engine handle the mixing in real time with no additional libraries required.

Streaming to Disk via File System Access API

A naive screen recorder would accumulate the entire recording in memory as a Blob and only offer a download at the end. For long recordings this easily consumes gigabytes of RAM and risks a browser crash. This recorder avoids that problem by streaming compressed video data directly to a file on disk.

When recording starts, window.showSaveFilePicker() opens the browser's native Save dialog. The user picks a location and filename, and the browser returns a FileSystemFileHandle. A FileSystemWritableFileStream is opened on that handle, giving the recorder a writable stream backed by the actual file.

The MediaRecorder emits compressed video chunks every second via its ondataavailable callback. Each chunk is immediately written to the writable stream with writableStream.write(chunk). The stream is closed — and the file finalised — only when recording stops. Memory usage therefore stays near-constant regardless of recording length.

This API is currently supported only in Chromium-based browsers (Chrome, Edge). Firefox has not implemented it, which is why the recorder requires Chrome or Edge.

Codec Selection

MediaRecorder supports different video codecs depending on the browser. The recorder tries the following MIME types in order of preference and uses the first one the browser reports as supported via MediaRecorder.isTypeSupported():

  1. video/webm;codecs=vp9,opus — VP9 video + Opus audio (best quality/size ratio)
  2. video/webm;codecs=vp8,opus — VP8 video + Opus audio (wider compatibility)
  3. video/webm — WebM with browser-default codecs
  4. video/mp4;codecs=h264,aac — H.264 video + AAC audio
  5. video/mp4 — MP4 with browser-default codecs

The detected MIME type is shown in the recorder settings panel so you always know what format will be produced. The file extension (.webm or .mp4) is set automatically to match.
