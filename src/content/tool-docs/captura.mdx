<div id="captura-about" class="card google-anno-skip">
  <div class="card-body">
    <h5 class="card-title">About Captura Web Recorder</h5>
    <p class="mb-2">
      A browser-based screen recorder inspired by the original
      <a href="/blog/2023/04/09/captura-unmaintained.html" target="_blank">Captura (no longer maintained)</a> desktop app.
      Record your screen, mix in webcam and microphone audio, and save the result — all entirely
      inside your browser. <strong>No data is ever sent to any server.</strong>
    </p>

    <h6 class="mt-3">Limitations vs. native desktop apps</h6>
    <ul class="mb-2">
      <li>
        <strong>No global keystroke/key overlay</strong> — browsers deliberately sandbox each tab
        and cannot listen to keyboard events outside the browser window. OS-level keyboard hooks
        (used by the original Captura to show keystrokes on screen) are simply not available to
        web pages. This is a fundamental security boundary of the web platform, and one of the
        reasons native desktop apps still shine for certain capture workflows.
      </li>
      <li>
        <strong>Desktop only</strong> — mobile browsers do not expose
        <code>getDisplayMedia</code> because the mobile OS sandbox prevents web pages from
        capturing other apps' screens. Use Chrome, Edge, or Firefox on a desktop.
      </li>
      <li>
        <strong>WebM or MP4 output</strong> — the recorder can produce either a
        <code>.webm</code> file (VP9 + Opus) or an <code>.mp4</code> file (H.264 + AAC) via the
        <a href="https://github.com/Vanilagy/mediabunny" target="_blank" rel="noopener noreferrer">Mediabunny</a>
        library (WebCodecs API). Select the desired format from the <em>Recording format</em>
        dropdown before starting. The choice is saved for future visits.
      </li>
      <li>
        <strong>File System Access API (Chrome/Edge) or OPFS (Firefox)</strong> — Chrome and
        Edge stream the recording directly to a folder you choose on your hard drive via the
        File System Access API; the save folder is remembered across visits. Firefox does not
        support the File System Access API, so the recorder falls back to the browser's
        <strong>Origin Private File System (OPFS)</strong>: the recording is staged in private
        browser storage and a download is triggered automatically when you stop recording.
      </li>
    </ul>

    <h6 class="mt-3">Hardware media key &amp; OS media overlay support</h6>
    <p class="mb-2">
      Once a recording is started, the browser registers a
      <a href="https://developer.mozilla.org/en-US/docs/Web/API/Media_Session_API" target="_blank" rel="noopener noreferrer">Media Session</a>
      so you can control it from <strong>hardware media keys</strong> (play/pause/stop buttons on
      keyboards, headsets, and remotes) or from the <strong>OS media overlay</strong> (e.g. the
      Windows taskbar media controls or macOS Now Playing widget) — even when the browser tab is
      not focused. The OS overlay shows the session as <em>"Captura Web – Recording Session
      Active"</em> and stays in sync as you pause, resume, or stop the recording.
    </p>

    <div class="alert alert-info mb-0 py-2">
      <i class="fas fa-shield-alt me-1"></i>
      <strong>Privacy:</strong> This tool runs 100% in your browser. No video, audio, or metadata
      is ever uploaded or transmitted anywhere.
    </div>

<div id="captura-how-it-works" class="card mt-4 google-anno-skip">
  <div class="card-body">
    <h5 class="card-title">How It Works</h5>
    <p class="text-muted small mb-4">
      A look under the hood for the technically curious.
    </p>

    <h6><i class="fas fa-desktop me-1 text-muted"></i>Screen &amp; Window Capture</h6>
    <p>
      Screen capture uses the browser's <code>navigator.mediaDevices.getDisplayMedia()</code> API.
      When you click <em>Start Recording</em>, the browser shows its own native picker where you
      choose what to share — an entire screen, a specific window, or a browser tab. The browser
      returns a <code>MediaStream</code> containing a video track (and optionally an audio track
      if "Share system audio" is checked in the picker). This is intentional: the browser is
      the gatekeeper, so no web page can silently capture your screen without your explicit
      permission each time.
    </p>
    <p>
      The system-audio checkbox in the recorder UI just passes <code>audio: true</code> as a
      constraint to <code>getDisplayMedia</code>. Whether system audio is actually available
      depends on the OS — Windows supports it natively; macOS requires a virtual audio driver.
    </p>

    <hr />

    <h6><i class="fas fa-th-large me-1 text-muted"></i>Canvas Compositor</h6>
    <p>
      Rather than recording the screen stream directly, the recorder draws every frame onto an
      offscreen <code>&lt;canvas&gt;</code> element. This intermediate step is what makes the
      webcam picture-in-picture (PiP) overlay possible: each frame, the compositor first draws
      the screen video, then draws the webcam video on top in the chosen corner, and finally
      stamps a live timestamp in the bottom-right.
    </p>
    <p>
      During recording the compositor runs in an async self-scheduling loop at your chosen frame
      rate (e.g. every ~33 ms for 30 fps). After drawing each frame it calls
      <code>canvasSource.add(timestamp)</code> from the Mediabunny library, which reads the
      canvas pixels and submits them to the hardware video encoder. The call is <code>await</code>ed
      so that if the encoder or disk I/O falls behind, the loop naturally slows down rather than
      overflowing an unbounded queue (back-pressure). The loop is a plain async function rather
      than <code>requestAnimationFrame</code> so that compositing continues reliably even when
      the browser tab is hidden or you switch to another application.
    </p>

    <hr />

    <h6><i class="fas fa-microphone me-1 text-muted"></i>Audio Mixing</h6>
    <p>
      System audio (from the screen capture) and microphone audio are two separate
      <code>MediaStream</code>s. To combine them into a single audio track, the recorder uses the
      Web Audio API:
    </p>
    <ol>
      <li>An <code>AudioContext</code> is created.</li>
      <li>Each source stream is wrapped in a <code>MediaStreamSourceNode</code>.</li>
      <li>Each source node is routed through a <code>GainNode</code>, whose gain value is bound
          to the corresponding slider in the <em>Audio Mix</em> panel. Moving a slider while
          recording instantly changes the mix without any interruption.</li>
      <li>Each gain node feeds into an <code>AnalyserNode</code> that is read every animation
          frame to drive the live level meters in the UI. The RMS amplitude of the time-domain
          samples is computed, scaled, and painted as a green/yellow/red bar — green up to 70 %
          of full scale, yellow from 70–85 %, and red above 85 % — so you can confirm your mic
          is live before committing to a long recording.</li>
      <li>Both analyser nodes connect to a single <code>MediaStreamDestinationNode</code>.</li>
      <li>The destination node exposes a mixed <code>MediaStream</code> whose audio track is then
          passed to Mediabunny as a <code>MediaStreamAudioTrackSource</code>.</li>
    </ol>
    <p>
      This approach lets the browser's audio engine handle the mixing in real time with no
      additional libraries required. Gain and mixer settings are persisted to
      <code>localStorage</code> so your preferred levels are restored on the next visit.
    </p>

    <hr />

    <h6><i class="fas fa-save me-1 text-muted"></i>Streaming to Disk</h6>
    <p>
      A naive screen recorder would accumulate the entire recording in memory as a
      <code>Blob</code> and only offer a download at the end. For long recordings this easily
      consumes gigabytes of RAM and risks a browser crash. This recorder avoids that problem by
      streaming compressed video data directly to a file as it is produced.
    </p>
    <p>
      <strong>Chrome / Edge (File System Access API):</strong>
      Before recording starts, the recorder asks for access to a <strong>folder</strong> of your
      choice via <code>window.showDirectoryPicker()</code>. The chosen folder handle is stored in
      <strong>IndexedDB</strong> so it persists across page reloads — you only need to pick a
      folder once. A <em>Choose Folder</em> button in the settings panel lets you switch to a
      different folder at any time.
    </p>
    <p>
      When recording starts, the recorder verifies it still has write permission for the stored
      folder using <code>FileSystemDirectoryHandle.queryPermission()</code>. If permission has
      lapsed (e.g. after a browser restart), <code>requestPermission()</code> is called to prompt
      you to re-grant it.
    </p>
    <p>
      <strong>Firefox (Origin Private File System fallback):</strong>
      Firefox does not support the File System Access API's directory picker. Instead, the
      recorder uses the <strong>Origin Private File System (OPFS)</strong> — a private,
      origin-scoped storage area exposed via <code>navigator.storage.getDirectory()</code>.
      No permission prompt is needed. The recording is staged in OPFS during capture; when you
      stop recording, a file download is triggered automatically so the video lands in your
      downloads folder. The temporary OPFS file is removed immediately after the download is
      initiated.
    </p>
    <p>
      In both cases, once a writable
      <code>FileSystemWritableFileStream</code> is open, it is passed directly to Mediabunny's
      <code>StreamTarget</code>, which writes encoded frames to the file incrementally as they are
      produced by the hardware encoder.
      When recording stops, <code>output.finalize()</code> flushes the encoder, rewrites the
      correct video duration into the file header, and closes the stream. Memory usage therefore
      stays near-constant regardless of recording length, and the output file always has a valid
      duration — unlike the truncated headers produced by the native <code>MediaRecorder</code> API.
    </p>

    <hr />

    <h6><i class="fas fa-keyboard me-1 text-muted"></i>Media Session API &amp; Hardware Media Keys</h6>
    <p>
      When recording starts, the recorder activates the browser's
      <a href="https://developer.mozilla.org/en-US/docs/Web/API/Media_Session_API" target="_blank" rel="noopener noreferrer">Media Session API</a>
      so that hardware media keys (play, pause, stop buttons on keyboards, headsets, and
      remotes) and OS-level media overlays (e.g. the Windows taskbar or macOS Now Playing
      widget) can control the recording even when the tab is in the background.
    </p>
    <p>
      Chrome on macOS (and most Chromium-based browsers) only forwards media session
      information to the OS — and only routes hardware key presses to the page — when a
      Core Audio session is open. Chrome opens a Core Audio session only when an
      <code>&lt;audio&gt;</code> or <code>&lt;video&gt;</code> element is playing with a
      non-zero <code>volume</code>. Additionally, <code>srcObject</code> live streams have
      infinite/unknown duration and are not forwarded to macOS Now Playing; a
      file-backed <code>src</code> with a finite, loopable duration is required.
    </p>
    <p>
      To satisfy both constraints without producing audible output, the recorder generates
      a minimal 100 ms silent PCM WAV in memory, exposes it via a Blob URL, and plays it
      on loop at <code>volume = 0.001</code> (−60 dBFS — 1/1,000th of full-scale amplitude,
      inaudible in any realistic environment). This opens a Core Audio session,
      registers the page with the macOS Now Playing widget, and activates hardware media
      key forwarding. The Blob URL is revoked and the element removed when the session ends.
    </p>
    <p>
      The handlers registered are <strong>play</strong> (resume a paused recording),
      <strong>pause</strong> (pause an active recording), and <strong>stop</strong>
      (stop the current recording). The OS media overlay is kept in sync by updating
      <code>navigator.mediaSession.playbackState</code> to <code>'playing'</code>,
      <code>'paused'</code>, or <code>'none'</code> whenever the recording state changes.
      The silent audio element and the Media Session metadata are cleaned up when the
      recording session ends.
    </p>

    <hr />

    <h6><i class="fas fa-film me-1 text-muted"></i>Codec &amp; Container (Mediabunny)</h6>
    <p>
      The recorder supports two output formats, selectable from the <em>Recording format</em>
      dropdown:
    </p>
    <ul>
      <li>
        <strong>WebM — VP9 + Opus:</strong> video encoded with <strong>VP9</strong>, audio with
        <strong>Opus</strong> at 128 kbps, muxed into a <strong>WebM</strong> container.
      </li>
      <li>
        <strong>MP4 — H.264 + AAC:</strong> video encoded with <strong>H.264 (AVC)</strong>,
        audio with <strong>AAC</strong> at 128 kbps, muxed into an <strong>MP4</strong> container.
        MP4 files are broadly compatible with most media players, devices, and video editors.
      </li>
    </ul>
    <p>
      Both formats are handled by the
      <a href="https://github.com/Vanilagy/mediabunny" target="_blank" rel="noopener noreferrer">Mediabunny</a>
      library, which drives the browser's hardware <code>VideoEncoder</code> and
      <code>AudioEncoder</code> (WebCodecs API) directly and writes the muxed output
      incrementally to the <code>FileSystemWritableFileStream</code> via its
      <code>StreamTarget</code>. On finalization it rewrites the duration metadata at the
      start of the file — a step that the native <code>MediaRecorder</code> API
      skips, making its output unscrubbable in many players.
    </p>
