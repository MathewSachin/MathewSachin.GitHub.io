*“We migrated and it feels faster” is not an audit. Performance claims only matter when they are measured, compared fairly, and tied to specific implementation choices.*

## Target query

What changed measurably after moving this site from Jekyll to Astro?

## Promise

You’ll get a repeatable audit framework focused on real outcomes: payload, rendering, and Core Web Vitals behavior.

## Step 1: Build a fair baseline

A fair before/after audit compares equivalent pages:

- one content-heavy post
- one homepage/listing page
- one interactive page

Measure each in similar network/device profiles and aggregate over multiple runs.

## Step 2: Attribute improvements to concrete changes

Typical migration wins came from specific decisions:

1. **Islands architecture** reduced unnecessary page-wide JavaScript.
2. **Typed content pipeline** reduced rendering mistakes and metadata drift.
3. **Asset handling improvements** made image/SVG delivery more predictable.
4. **Simpler build path** reduced accidental regressions in deployment output.

Without attribution, metrics are trivia.

## Image and SVG handling impact

Image strategy affects LCP more than most teams expect.

Important checks:

- LCP element format and dimensions
- responsive image behavior
- SVG processing policy consistency

Migrating stack is a chance to enforce better defaults instead of carrying historical debt.

## Hydration budget: the hidden lever

For blog-like sites, hydration budget is often the biggest practical win.

Audit each template:

- What must hydrate immediately?
- What can hydrate on idle/visibility?
- What should remain static forever?

JavaScript not shipped is the fastest JavaScript.

## Measurable outcomes to track monthly

Use a compact scorecard:

- LCP median and p75
- CLS p75
- Total JS transferred per template
- Largest route HTML size
- Third-party script cost

A migration is not a one-time optimization event; it should establish a recurring performance discipline.

## Examples/screenshots to include

- Baseline vs post-migration Lighthouse summary
- JS payload breakdown before/after
- Hydration map by page template
- Web Vitals trend chart over time

## Actionable takeaway

Set explicit page-template budgets (JS bytes, LCP target, third-party cost), and block changes that exceed them unless they ship deliberate user value.
