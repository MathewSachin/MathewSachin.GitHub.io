*“It feels faster” is a useful intuition, but it is not evidence. After migrating from Jekyll to Astro, I wanted to know which changes actually moved numbers and which ones merely felt modern.*

A good audit answers two questions: what improved, and why.

## Build a fair before/after comparison

Meaningful comparisons need equivalent page types—content-heavy posts, listing pages, and interactive routes—under similar network/device profiles.

When the baseline is fair, you can attribute gains with confidence instead of cherry-picking a single route that flatters the new stack.

## Where the largest gains usually came from

In this migration, the biggest practical improvements were tied to architecture choices, not “micro-optimizations”:

- less unnecessary JavaScript from islands-first rendering
- cleaner content pipeline with typed metadata
- predictable static output and asset paths

These changes reduced both payload and regression risk.

## Hydration discipline matters more than most tweaks

On content-heavy sites, hydration budget is often the hidden lever. Every component that doesn’t need to hydrate is one fewer script competing for main-thread time.

Template-by-template hydration decisions usually outperform random bundle-level tuning done later.

## Image and SVG policy still affects real metrics

Even with a strong static stack, media handling can dominate LCP behavior. Route-level audits should still check:

- which element is LCP
- whether media dimensions and format choices are appropriate
- whether asset processing policy remains consistent

Performance regressions often re-enter through content and media changes, not framework code.

## Treat performance as an operating practice

Migration is a reset point, not the finish line. I’ve found it more effective to keep a small recurring scorecard—LCP, CLS, JS transferred, largest HTML payload, third-party cost—than to run occasional one-off audits.

That rhythm makes regressions visible early and keeps optimization tied to real user impact.
