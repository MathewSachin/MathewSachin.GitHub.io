*People ask one question about browser PDF tools: “Are you uploading my file?” If your product can’t answer that clearly and truthfully, trust is gone before the first click.*

## Target query

How can a PDF password tool add or remove passwords entirely in the browser while staying safe and usable?

## Promise

This post explains the browser-side architecture, security model, and UX decisions behind a practical PDF password tool.

## Processing model: local execution via WASM + browser APIs

At a high level:

1. user selects a PDF file
2. file bytes are loaded into browser memory
3. WebAssembly-powered PDF logic reads/modifies encryption metadata
4. new PDF bytes are generated
5. user downloads processed output

No backend upload is required for the core operation.

## Add-password flow

A robust add-password flow should include:

- file validity checks
- password strength guidance
- confirmation for typo prevention
- output filename clarity (`*-protected.pdf`)

UX detail that matters: users need explicit confirmation that original file is untouched.

## Remove-password flow

Removing a password is usually trickier:

- input may use unsupported encryption variants
- wrong password handling must be clear
- errors should distinguish “bad password” from “corrupt file”

A generic “failed” message creates support churn and user distrust.

## Privacy and security claims: what to say precisely

Safe and accurate language:

- “Processed locally in your browser”
- “No server upload required for conversion”
- “Close tab to clear in-memory state”

Avoid overclaiming with absolute phrases you cannot verify in every environment.

## Edge cases you should design before launch

- very large files (memory spikes)
- malformed PDFs
- already-protected files with unknown methods
- user canceling mid-process
- browser tab suspension/throttling

If these are ignored, happy-path demos look great but real users fail.

## UX decisions that improve trust

- show clear stage labels (loading, validating, processing, done)
- show explicit local-processing note near upload button
- provide actionable error messages
- keep controls accessible for keyboard and screen reader users

## Examples/screenshots to include

- Add-password flow with validation states
- Wrong-password error state
- Large-file warning/handling state
- Privacy explainer callout near upload

## Actionable takeaway

Design error states first, then happy path. In document-security tools, clear failure handling is a core feature.
