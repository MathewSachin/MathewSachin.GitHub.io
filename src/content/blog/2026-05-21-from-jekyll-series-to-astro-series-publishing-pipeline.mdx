*The migration is only half done when pages render in a new framework. It is fully done when your publishing workflow becomes easier, safer, and faster than before.*

## Target query

How do I evolve a mature Jekyll publishing workflow into an Astro-native content pipeline?

## Promise

This post gives a practical blueprint for day-to-day publishing in Astro: content schema, routes, previews, and deployment discipline.

## Content collections as the new foundation

In Jekyll, frontmatter drift can hide for weeks.

In Astro, typed collections let you validate metadata at build time.

Benefits:

- required fields enforced early
- optional fields documented in one place
- fewer silent formatting mistakes

Treating metadata as schema is a huge reliability upgrade for long-running blogs.

## Route shape consistency during migration

Users and search engines expect URL stability.

If legacy posts used date-based routes, preserve route shape while modernizing internals.

Good migration behavior:

- keep canonical URL patterns stable
- maintain redirects where route changes are unavoidable
- validate internal links after content moves

## BASE_URL and preview pitfalls

Preview environments often run under subpaths. Hardcoded root-relative links break quietly.

Adopt a strict policy:

- internal links/components should be BASE-aware
- test previews using the same base-path behavior as CI

This one detail prevents a large class of “works locally, broken in preview” defects.

## New publishing flow (draft to live)

A reliable Astro-era flow can be:

1. create dated `.mdx` post
2. write with schema-valid frontmatter
3. build locally and run tests
4. open PR with preview link
5. review content + UX + links
6. merge and deploy

Automating parts of this flow is useful, but consistency matters more than automation quantity.

## Organizing future series content

Moving from a Jekyll-era series to Astro-first series is a chance to define clearer editorial structure:

- standalone deep dives for high-intent search queries
- linked series hubs for topic continuity
- consistent tag clusters to improve discovery

This makes content strategy and technical architecture reinforce each other.
