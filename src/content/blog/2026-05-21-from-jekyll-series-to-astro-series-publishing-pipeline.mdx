*Framework migration is only half the story. The real win is when the publishing workflow becomes more reliable for future posts than the old one ever was.*

Moving from Jekyll-era habits to an Astro-native pipeline changed how I write, review, and ship content day to day.

## Typed content changed editorial reliability

In older workflows, frontmatter mistakes could stay invisible until a page looked wrong in production. With typed collections and schema validation, metadata issues fail early during build.

That shifts quality control from “spot later” to “block before merge,” which is exactly where content correctness should sit.

## URL continuity is a migration requirement

Readers and search engines care about stable URLs more than internal framework choices.

During migration, preserving route shape and handling redirects where needed prevents avoidable SEO and navigation regressions. It also keeps historical links from old posts and social shares useful.

## BASE-aware linking prevents preview surprises

One subtle but recurring source of issues is base path handling in previews. Root-relative links can look fine locally and fail in preview deployments.

Adopting BASE-aware internal linking as a strict convention removed a class of “works here, broken there” bugs from the publishing flow.

## The new day-to-day flow is simpler

The current flow is intentionally boring:

1. create dated `.mdx` file
2. write schema-valid frontmatter + content
3. run build/tests/checks
4. open PR and review via preview
5. merge only when content and navigation pass

This consistency matters more than adding more automation scripts.

## Series strategy also got clearer

Migration was a good forcing function to tighten editorial structure: deeper standalone posts for search intent, better internal linkage between related posts, and cleaner tag clusters for discovery.

In practice, technical architecture and content strategy now reinforce each other instead of fighting each other.
