*Search relevance does not emerge automatically from “index everything.” It is a sequence of deliberate choices: what to include, what to strip, what to boost, and how to evaluate outcomes against real user queries.*

## Target query

How should I build and tune static-site search in Astro so results are actually useful?

## Promise

You’ll leave with a practical model for index generation, relevance tuning, and debugging search misses.

## Index generation flow

A robust static search pipeline typically does this:

1. collect content sources (posts, tags, tools, series pages)
2. normalize metadata (title, URL, tags, category)
3. strip markdown syntax/noise from body text
4. serialize to a search-engine-friendly JSON index
5. ship index with static build

This keeps runtime search fast and infrastructure-free.

## What to index (and what not to)

Good search quality often improves when you **exclude** noisy content.

Include:

- title
- concise description
- tags
- meaningful body text

Exclude or downweight:

- code fences (unless code search is a feature)
- boilerplate nav/footer text
- repeated CTA copy

The goal is query intent alignment, not maximum token count.

## Relevance tuning strategy

Start with a simple weighting model:

- title: highest boost
- tags: medium-high
- description: medium
- body: baseline

Then tune against actual query logs.

### Practical tuning loop

1. Pull top queries from analytics/search logs
2. Label expected top results for each query
3. Run offline relevance checks after each adjustment
4. Compare precision of top 3 results
5. Ship only improvements that help real queries

## Debugging bad matches quickly

When a result looks wrong, inspect:

- was the page indexed?
- was content stripped too aggressively?
- did tags overpower semantic relevance?
- is stale index being served from old build?

Most “search is broken” incidents are stale-index or weighting mistakes, not engine bugs.

## Common pitfalls in static search

- rebuilding pages but forgetting to regenerate index
- over-indexing long body text so niche keywords dominate
- relying on synthetic queries instead of user behavior
- not testing search after URL/routing changes

## Actionable takeaway

Maintain a living benchmark set of your top 20–50 real queries, and tune search against that benchmark every time you change extraction or weighting.
