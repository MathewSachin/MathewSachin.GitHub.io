*The easiest way to build search is to index everything and call it done. The easiest way to get bad results is also to index everything and call it done.*

Search quality on static sites comes from choices: what to include, what to strip, what to boost, and how to validate ranking against real user intent.

## Index generation is where relevance begins

For static search, most important decisions happen before runtime:

- choose source documents (posts, tags, tools, series pages)
- normalize metadata consistently
- strip markdown noise from content bodies
- serialize a compact index and ship it with the build

Runtime querying is fast only because this preprocessing does the hard work up front.

## Better search often means indexing less

Over-indexing is one of the most common self-inflicted problems.

Title, description, tags, and meaningful body text are usually enough. Boilerplate UI text, repetitive CTA copy, and irrelevant code fences can dilute results and push wrong pages higher than they should be.

The goal is not maximum token volume; the goal is signal density.

## Relevance tuning should follow real queries

I start with intuitive field weighting—titles highest, tags high, description medium, body baseline—then tune using observed user queries.

A practical loop looks like this:

1. collect top real queries
2. define expected top results
3. run comparisons after each weighting change
4. ship only if top results improve materially

Without this loop, relevance tuning becomes guesswork with fancy terminology.

## Debugging “search is broken” quickly

When rankings look wrong, I check simple causes first:

- was the page included in the index?
- was content stripped too aggressively?
- are tags overpowering body relevance?
- is an old index being served from a stale build?

In practice, stale index artifacts and extraction mistakes are far more common than engine-level bugs.

## Keep a relevance benchmark as part of maintenance

Search quality drifts when content changes, URLs evolve, and new taxonomies are introduced. A small benchmark set of high-intent queries catches drift early and makes tuning safer.

Treat that benchmark like a test suite for relevance.
