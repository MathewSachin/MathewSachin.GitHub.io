*When people say “Playwright is flaky,” I usually hear “our tests are making assumptions the browser does not guarantee.” Static sites have fewer moving parts than full apps, but they still have hydration, async index loads, and browser permission edges.*

The good news is that reliability improves a lot once you model those realities explicitly.

## Most flakiness comes from readiness assumptions

The most common anti-pattern is treating `page.goto()` as “the page is ready for all interactions.”

For each page, define a concrete readiness signal before you click anything. On search pages, maybe that means the input is enabled and loading status is gone. On tool pages, maybe it means a primary action button is visible and enabled.

Once that first readiness assertion is explicit, a huge class of random failures disappears.

## Keep retries narrow and intentional

Retries are useful for interactions that race with hydration or browser events. They are harmful when used as a blanket at the test level.

I use bounded retries for things like:

- file chooser timing windows
- clipboard reads after async copy actions
- a click that occasionally races with late-rendered markup

I avoid retries for final behavior assertions. If the end state is wrong, I want the test to fail loudly.

## Test the artifact users actually receive

Static sites can behave differently between dev server and built output. That’s why the stable pattern is:

1. build the site
2. serve `dist/`
3. run e2e against that output

It catches broken paths, stale assets, and index delivery issues that local dev mode can hide.

## Debugging should be fast, not forensic archaeology

A failing CI run should be diagnosable in minutes. For that, artifacts matter as much as assertions.

At minimum, keep:

- Playwright trace
- screenshot/video where relevant
- concise logs tied to failing step

When this is in place, triage becomes “open trace, inspect transition, patch assumption,” not “rerun and guess.”

## Selector quality is part of test reliability

Tests should track user intent, not CSS implementation details. IDs, roles, and stable attributes age better than long nested selectors.

Whenever a test starts failing after a harmless refactor, it’s often a sign the selector targeted styling structure instead of user-facing semantics.

## The practical checklist I use now

Before merging a new e2e test, I ask:

- does the test define explicit page readiness?
- does it avoid fixed sleeps?
- are retries scoped to race-prone interactions only?
- could someone debug a failure from stored artifacts alone?

If yes, that test usually survives CI load and future refactors.
