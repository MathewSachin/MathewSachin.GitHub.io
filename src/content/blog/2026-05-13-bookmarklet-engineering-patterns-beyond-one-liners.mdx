*The internet loves bookmarklet one-liners. I’ve written and used plenty of them too. The problem is that one-liners are usually optimized for “looks clever in a tweet,” not “keeps working next month.”*

If you’ve ever shared a bookmarklet, then gotten a DM saying “it stopped working,” you already know the real challenge isn’t writing JavaScript. It’s dealing with pages that change every week.

## The failure pattern is boringly consistent

Almost every broken bookmarklet I debug ends up failing for one of the same reasons:

- the selector depended on fragile class names or DOM position
- the script ran before the target UI finished rendering
- there was no fallback when the first query returned nothing
- errors failed silently, so users only saw “nothing happened”

That combination makes bookmarklets feel random, even when the original idea is good.

## Think in strategies, not snippets

The biggest shift is mental: a bookmarklet should be treated as a tiny product, not a throwaway snippet.

For element targeting, I start with meaning-first selectors (`id`, `data-*`, ARIA labels, button text) and keep class-based selectors as fallback only. Then I design a lookup chain, not a single query. If method A fails, try B. If B fails, try C. If all fail, report clearly what was attempted.

That one change turns “mysterious breakage” into “diagnosable behavior.”

## Timing is part of correctness

Modern pages hydrate in phases. A bookmarklet that assumes “click means DOM is ready” will fail intermittently on real-world connections.

A reliable flow is:

1. attempt immediately
2. retry after a short delay
3. optionally use a bounded observer when content is rendered late

The key word is *bounded*. Infinite polling is just another failure mode.

## Mobile is where many bookmarklets quietly die

Desktop Chrome success does not imply mobile success.

Invocation behavior differs across browsers, some interactions depend on hover/right-click, and not every mobile browser exposes the same UX for bookmarks. If a bookmarklet is meant to be shared, I test at least Android Chrome and iOS Safari before calling it “done.”

## Know the boundary of what bookmarklets can do

Bookmarklets are great for client-side convenience: cleanup, copy helpers, visibility toggles, quick transformations.

They are not a tool for bypassing real server-side controls. CSP rules, iframe boundaries, and backend authorization still exist for a reason. Designing with those boundaries in mind leads to better tools and less confusion.

## A practical release bar

Before publishing, I ask a few boring but useful questions:

- Do I have at least one fallback selector path?
- Does it fail safely when the target is missing?
- Is there a clear error message for users?
- Did I validate on mobile as well as desktop?
- Can I explain exactly where it is expected *not* to work?

If the answer to any of these is no, it’s still a draft.
